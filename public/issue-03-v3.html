为什么资深开发者讲不清自己的专业能力？

2026.05.15 · X：宝玉

一篇很有意思的长文。作者说，业务团队活在"消除不确定性"的循环里——他们要的是快、是验证。而资深开发者活在"管理复杂性"的循环里——他们要的是稳、是不崩。两边语言不通，所以开发者说"这个需求会增加系统复杂度"，业务听成了"他不想干"。

为什么值得关注？

  • AI 能写代码，但没法"承担责任"。文章最后说了，资深开发者不可替代的价值在于：系统塌了的时候，有人能兜底。AI 没有背锅能力，这是它和人之间最本质的差距。
  • 对小白用户的启示。你不需要成为"资深开发者"才能用 Claude Code。你要学的是怎么把任务拆清楚、怎么验收结果——这些能力，业务思维和工程思维都需要。

三条新闻的共同点：都在说同一件事——AI 正在从"能说"变成"能干"。但"能干"的前提是安全、可控、有边界。普华永道用 Claude 干专业的事，Codex 用沙箱保安全，开发者用经验兜底。对咱们来说，学会用 Claude Code，就是学会给 AI 设边界、下指令、验结果。

02

📚 概念 & 案例

Claude Code 是怎么把 Harness"落地"的

上期我们聊了 Harness 的 5 个组件——环境感知、文件操作、终端执行、多步规划、工具调用。很多朋友看完还是懵：这些组件在 Claude Code 里到底长什么样？

我跟你说，Claude Code 就是把这 5 块拼图串起来的一个具体产品。它不是 Claude 的网页版，它是 Claude 的"动手版"。今天一个一个拆开给你看。

一、Claude Code 到底是什么？

先做个类比：

普通对话 AI 就像一个电话客服。你问他怎么修水管，他说："先关水阀，再拧螺丝..." 你挂了电话，自己翻工具箱、找扳手、拧到一半发现尺寸不对。

Claude Code 就像一个带着全套工具的维修师傅上门。你说"水管漏了"，他自己找水阀、拧螺丝、测压力，修完还拍照给你确认。

差别就在 Harness——有没有手。

二、5 个组件在 Claude Code 里的样子

上期讲了概念，这次我直接给你看"实物"：

1
环境感知——你打开 Claude Code，它自动扫描当前文件夹里的所有代码，理解项目结构。你不需要手动复制粘贴代码给它，它自己看。
2
文件操作——你说"新建一个 about.html"，它直接创建文件、写入内容、保存。不需要你来回切换窗口复制粘贴。
3
终端执行——它能运行 npm install 装依赖、启动本地服务器、跑测试脚本。你之前要在终端里一行行敲的命令，它帮你执行。
4
多步规划——你说"帮我做个个人博客"，它自动拆成：建文件夹 → 写首页 → 写样式 → 启动服务器 → 打开浏览器。一步接一步，不用你催。
5
工具调用——通过 MCP 接口，它能查新闻、操作数据库、发邮件。就像给 Claude Code 装 App，装了就能干更多事。

一句话总结：1-3 是"手"（能动手干），4 是"脑"（知道先干啥后干啥），5 是"外挂"（能干的事无限扩展）。

三、安全问题：它会不会把我电脑搞坏？

这是我最常被问到的问题。坦率的讲，完全不管让 AI 在电脑上随便跑，确实有风险。但 Claude Code 有三道保险：

第一道：执行前确认。所有涉及文件修改、命令执行的敏感操作，Claude Code 都会先问你"我要执行这个，可以吗？" 你不点头，它不动。

第二道：黑白名单。你可以设置"允许操作的范围"，比如只允许修改某个文件夹，其他地方的文件一律不动。

第三道：完整日志。它干了什么、改了哪个文件、跑了什么命令，全部有记录。出了问题你可以回溯。

你想想看，这比很多真人助理还靠谱——至少它不会背着你偷偷干。

四、和 Codex CLI 的区别

上期我们提到 Codex CLI 是 Claude Code 的竞品，这期它出了 Windows 沙箱。两款工具核心差别在哪？

Codex CLI 更像一个"超级自动补全"——你在写代码的时候它帮你补、帮你改、帮你跑测试。它的沙箱是"隔离式安全"：AI 被关在一个小房间里，随便折腾，但出不了门。

Claude Code 更像一个"项目经理"——你给它目标，它自己规划步骤、协调工具、推动执行。它的安全是"许可式安全"：每一步都先问你，你才是最终拍板的人。

你选哪个？如果你主要是写代码，Codex 更顺手；如果你经常需要"从零搭一个东西"，Claude Code 更省心。

上面讲的都是理论，可能你还是没感觉。现在咱们来一个真实对比——同一个需求，三种工具分别怎么回应。

Claude Code 的本质就是 Harness 的一个具体落地产品。它不改变 Claude 模型的"智商"，但把这个智商变成了一双能干活的手。

💡 灵魂洞察

不只关于技术，关于人与工具

普华永道不是被 Claude 替代了 30 万员工，

而是这 30 万人每人多了个能干活的助理。

问题从来不是"AI 会不会让我失业"，
而是"我会不会让 AI 替我思考"。

我跟你说个 analogy（类比）。Claude Code 就像一把电钻。

你不会因为有了电钻就变成木匠。但你如果会用电钻，你打孔的速度就是手摇钻的十倍。关键是——你知道要在哪里打孔。

今天那篇"资深开发者讲不清专业能力"的文章，其实讲的就是这个。业务团队问的是"能不能快点"，开发者答的是"系统会不会崩"。两边都没错，但两边说的不是一件事。

AI 的出现，其实给了第三种可能：你不需要成为资深开发者，也能做出不错的东西。 Claude Code 把"写代码"这件事的门槛，从"学三年"降到了"说清楚需求"。

但这里有个坑，我得提醒你。

很多人用 Claude Code 的时候，会犯一个错误：把思考也外包给 AI。"帮我做个网站"——然后 AI 怎么做你完全不管，最后出来的东西不满意，你又怪 AI 不行。

说真的，AI 是杠杆，不是替身。杠杆放大的是你的方向感。方向对了，十倍速；方向错了，十倍错。

普华永道的员工为什么能把交付时间压 70%？不是因为他们把脑子丢了全交给 AI，而是他们知道"这个任务的目标是什么、验收标准是什么、哪里可能出问题"。AI 负责执行，人负责判断。

所以回到那个问题：用 AI 多了会不会变笨？

我的感受是，关键不在"用不用"，而在"怎么用"。 你把 AI 当"代笔者"，脑子确实会退化。你把 AI 当"电钻"，手会越来越稳。

模型是电钻，Harness 是电源，你才是那个握钻的人。握钻不需要博士学历，需要的是知道"我要做个什么东西"。
